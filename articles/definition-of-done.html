Definition of done

By Winda Sekar Dewi

Overview

  1. When the Definition of Done is complete, it defines all steps needed to deliver a finished increment, creating feedback on both the product and the process within the sprint.
  2. One purpose of the DoD is to guarantee that things work well and meet basic technical requirements. It is typically led by the Scrum Master or the head of engineering, but it should be a collaborative exercise. Without input and approval from product, QA, and other stakeholders, there will not be widespread acceptance of what "done" actually means.
  3. Meeting the DoD ensures quality. It lowers rework by preventing user stories that do not meet the definition from being promoted to higher-level environments, and prevents features from being delivered to users before they are truly ready.

Definition of done in capacity estimation

Projects with a development effort of more than 6 weeks (considered large-sized in capacity estimation) can be divided into several projects to have clearer delivery visibility.

Criteria Chunk-able Releasable Able to deliver value
Deliver value partially YES YES YES
Release (not yet rolled out) partially YES YES NO
Big bang release YES NO NO

Deliver value partially

The project can be divided into several releases based on acceptance criteria, without breaking the user flow. Example: Active/Inactive Warehouse:

  • Phase 1: Index and details: user can inactivate a warehouse, view inactive warehouse list, and see all warehouses in select dropdowns.
  • Phase 2: Validation in transactions: validation of inactive warehouses in inventory, sales, and purchase transactions.

Released or rolled out partially

Projects that cannot be divided by user flow can be split by module with a feature toggle. Each module can be released to staging or production and toggled on for specific users during testing. Once tested and agreed upon by all stakeholders (engineering, QA, product, management), the module can be marked as done.

Sprint goal in sprint

After capacity estimation, it is ideal to have more than 80% of the PRD, RFC, engineering PIC, resource count, and task breakdown ready before sprint planning. This allows creation of a detailed timeline based on initiative priority and mapping of each initiative to specific sprints in the quarter.

Example: an initiative with 8 weeks of total development and testing effort, where everything must be done before testing and rollout:

  • Sprint 1: Tasks A–C in week one, Tasks D–F in week two. Sprint goal: Dev Done. Conduct sprint review and gather feedback.
  • Sprint 2: End-to-end testing of Tasks A–F. Bug fixing (may carry over to week one of the next sprint). Sprint goal: Bug Fix Dev Done.
  • Sprint 3: Continue bug fixes. Retest fixed bugs. Sprint goal: Passed QA.
  • Sprint 4: Deploy to production and roll out.
Common sprint goal stages used in practice:
  • Dev Done (FE and/or BE without integration)
  • Passed QA (tested on staging)
  • Released without rolling out
  • Rolled out
  • RFC ready to review
  • RFC passed review
