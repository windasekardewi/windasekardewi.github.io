Monday sync used to cost me hours

By Winda Sekar Dewi

Every Monday morning, I had one job before the sync started: figure out which of our active initiatives was on track, which one was quietly on fire, and what someone needed to decide before it got worse.

For a long time, that meant opening Jira, clicking through epics one by one, cross-referencing notes, and assembling a Confluence update by hand. About an hour of work. Accurate at 9 AM, stale by 11.

I got tired of that. So I built something better.

I packaged an AI-assisted workflow (I call it a "skill") that reads live project data and produces a program health report automatically. It does not just count tasks by status. It reads through the comment threads on every epic and child task, looking for things ticket status never shows you: a blocker mentioned in passing, a decision deferred mid-thread, an engineer flagging something without ever moving the ticket. It also cross-references commit activity and PR review age from the code repository, so if a ticket has been "In Progress" for a week with no code moving, that surfaces too.

The output is ready before the sync starts: initiative name, team, status, progress percentage, target date. Risk flags with a named owner and the specific action needed. An actions table so everyone knows what they need to decide before next week.

What the skill does, technically

Six steps under the hood:

  1. Takes the program or epic key as input.
  2. Queries all child epics via the project tracker API.
  3. Fetches all child tasks per epic, counts by status, then deep-dives into comments on both the epic and each task. Looking for blockers mentioned in passing, decisions deferred mid-thread, engineers flagging issues without moving a ticket.
  4. Pulls code repository data: PR counts, review age, and commit activity per initiative, cross-referenced against what the tracker says is happening.
  5. Applies detection rules: stuck tickets (no status change in N days), keyword-scanned blockers in comments and transitions, PRs sitting in review past a set threshold, unassigned tasks on active epics.
  6. Writes the full report to the team wiki.

What actually changed

  • Sync prep time: from about an hour to about 15 minutes.
  • Every risk flag has a named owner. "Will be followed up" is gone.
  • I stopped asking "what is the status?" I started asking "what needs to be decided before this gets worse?" That is a very different kind of meeting.
  • Onboarding to a new team used to take weeks. The same skill helped me read multiple quarters of delivery history, incident patterns, and team context before my first sync. I was ready in days, not weeks.

What it did not fix

The output is only as good as the data. Epics with no child tasks, initiatives with no delivery date, tickets that have never had a status change: all of that creates gaps. Honestly, one of the most useful things this skill did was show me exactly where our data discipline was weak.

It also does not replace judgment. The skill can flag that an initiative is at 30% with three weeks left. It cannot know whether the team already agreed to descope it last Friday. That kind of context still lives with people. The artifact creates structure for the conversation, not the conversation itself.

If you want to build this for your team

The version I built is wired to my specific context. If you are adapting it, here is what you will actually need to change.

Your team structure and naming

The skill needs to know how your org names things: squad names, team labels, initiative types. If your project tracker has custom fields for health status, delivery dates, or criticality, those need to be connected. Better yet, set up the skill to discover them from field metadata before it starts pulling data, so you are not hardcoding IDs that break when someone renames a field.

Your toolchain

I use Jira and Bitbucket, with output going to Confluence. If your team runs on GitHub and Notion, or Linear and Slack, the connectors change. The logic does not.

Your thresholds

"Stuck after 7 days" works for my team's cadence. A team on 1-week sprints might flag at 4 days. A team doing continuous delivery might not use sprints at all. The PR review age threshold, the velocity formula, the overdue logic: all of it should reflect how your team actually works, not a generic default.

Your hierarchy

Some orgs are flat: epics and tasks. Others have programs containing epics containing stories containing subtasks. Where the skill enters and how deep it goes depends on where your "program" actually lives in that structure.

Your audience

A report for engineers should be precise and short. A report for a VP or a business stakeholder needs more narrative and less tracker jargon. The recommended actions section especially changes depending on who is reading. Build the writing style into the skill so you are not editing every output before you send it.

The first version does not need to be perfect. It just needs to exist, so you can stop spending an hour every Monday doing something a machine can do in two minutes.
